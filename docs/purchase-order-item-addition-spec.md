# 規格：採購確認時追加任意商品規格

## 已確認需求與假設

1. 使用者是具有 `採購確認` 或 `管理員` 角色的採購者。
2. 採購者可在第一次確認下單前，從 `SKU主檔` 中全部「是否可補貨＝是」的商品／規格任意追加，不限原申請商品。
3. 原申請品項仍可調整下單數量；數量設為 0 代表該品項取消採購。
4. 追加品項沿用原補貨單號，申請數量記為 0，下單數量記錄採購者實際追加量。
5. 確認下單後不可再修改或追加；若日後需要修改已下單資料，另開規格處理。
6. 不新增 Google Sheets 欄位、不修改飛鼠後台，也不新增第三方套件。

## 目標

讓老闆在 LINE LIFF 的「確認下單」頁面中，除了調整原申請品項數量，還能搜尋全部可補貨商品、選擇原申請沒有的規格並加入同一張補貨單。確認後，原品項與追加品項必須一起出現在 LINE 下單通知、到貨確認與後續查詢中。

## 使用流程

1. 採購者開啟待確認補貨單。
2. 頁面顯示原申請品項及可調整的下單數量、預計到貨日。
3. 採購者點擊「追加商品／規格」，可搜尋全部目前可補貨的商品、規格或 SKU。
4. 採購者選擇一個或多個規格及數量；已在本單中的 SKU 顯示為已加入，不可重複新增。
5. 追加品項回到確認清單後標示「採購追加」，可調整數量或在送出前移除。
6. 採購者確認下單後，系統一次寫入原品項更新、追加品項新列及操作紀錄。
7. LINE 群組通知列出實際下單品項；後續到貨確認包含所有下單數量大於 0 的原品項與追加品項。

## 資料與驗證規則

- 後端必須保留全部原申請 SKU；原品項不購買時需明確提交下單數量 0，不可從請求中省略。
- 追加 SKU 必須在送出當下仍為「是否可補貨＝是」，不可只相信前端先前載入的資料。
- 同一張補貨單內 SKU 必須唯一。原品項與追加品項相同時，沿用原列調整數量，不建立重複列。
- 原品項下單數量允許 0；追加品項下單數量必須是 1 以上的整數。
- 每張補貨單確認時合計最多 50 個唯一 SKU；單一品項最大數量沿用前端既有限制 999999。
- 下單數量大於 0 時必須填寫合法的 `YYYY-MM-DD` 預計到貨日。
- 只有補貨單所有既有列皆為 `待確認` 時才能確認；重送使用既有操作金鑰回傳相同結果，不重複新增列。
- 新增列沿用原補貨單號、申請時間、申請人、原始備註、申請人 LINE ID 及來源群組；`最後操作人ID` 記錄實際採購者。
- 新增列的 `申請數量` 為 0、`狀態` 為 `已下單`、`已到數量` 為 0，並沿用既有公式欄與資料格式。
- 追加列只附加到 `補貨追蹤` 最後一筆資料之後，不覆蓋中間空白列後方的既有資料。
- 原品項更新、追加列與 `操作紀錄` 使用同一批 Google Sheets 寫入；任一驗證失敗時不得部分寫入。

## 技術架構

- Runtime：Node.js 22+、ESM。
- HTTP：Express 5。
- 前端：原生 HTML、CSS、JavaScript 與 LIFF SDK。
- 資料：Google Sheets API；沿用 `補貨追蹤`、`SKU主檔`、`商品圖片對照` 與 `操作紀錄`。
- LINE：沿用 Messaging API、LINE Login 與既有角色驗證。
- 商品選擇：沿用 `public/catalog.js` 的商品分組、搜尋、規格名稱及圖片回退規則。

## Commands

- 開發：`npm.cmd run dev`
- 指定測試：`node --test test/workflow.test.js test/sheets-repository.test.js test/api.test.js test/catalog.test.js test/messenger.test.js`
- 完整測試：`npm.cmd test`
- 語法檢查：`npm.cmd run lint`
- 建置檢查：`npm.cmd run build`

## Project Structure

- `src/services/workflow.js`：角色、數量、日期、SKU 唯一性與品項上限驗證。
- `src/sheets/repository.js`：重新驗證可補貨 SKU、更新原列、附加追加列及操作紀錄。
- `public/app.js`：確認下單頁與追加商品選擇流程。
- `public/catalog.js`：可重用的商品分組、搜尋與規格工具。
- `public/index.html`、`public/styles.css`：手機版追加商品介面。
- `src/line/messenger.js`：下單結果通知。
- `test/`：服務、repository、API、商品工具與訊息測試。

## Code Style

沿用現有具名匯出、早期驗證及可讀錯誤訊息：

```js
if (items.length > MAX_ORDER_ITEMS) {
  throw new ValidationError(`單張補貨單最多可下單 ${MAX_ORDER_ITEMS} 個品項`);
}
```

前端不直接決定資料是否合法；所有 SKU、角色、狀態與數量仍由後端重新驗證。

## Testing Strategy

- 服務測試：角色限制、50 項上限、SKU 重複、數量與日期正規化。
- Repository 測試：允許合法追加、拒絕停用或不存在 SKU、原品項不可省略、追加列欄位正確、操作重送不重複寫入、容量不足整批停止。
- API 測試：確認下單 payload 可包含追加 SKU，並維持 LINE 身分與角色路徑。
- 商品工具測試：全商品搜尋、已加入 SKU 排除與購物清單計數。
- 訊息測試：下單通知包含追加品項，取消品項不列入實際下單清單。
- 手機驗收：搜尋任意商品、追加不同商品規格、調整原品項、確認下單、查看群組通知及到貨頁。

## Boundaries

### 一定會做

- 後端重新驗證角色、補貨單狀態、SKU 可用性與唯一性。
- 使用冪等操作金鑰，避免雙擊造成追加列重複。
- 保留原申請資料並記錄實際採購操作人。
- 不影響既有不追加商品的確認下單流程。

### 需要另行確認

- 部署至正式 Cloud Run。
- 修改正式 Google Sheet 結構或欄位公式。
- 允許已確認下單後再次編輯或追加。

### 不會做

- 不讓一般申請人使用採購追加功能。
- 不加入停用、待確認或不存在的 SKU。
- 不直接修改飛鼠庫存或商品資料。
- 不把前端傳入的品名、單位或價格當成可信資料。

## Success Criteria

1. 採購者能在確認下單頁搜尋全部可補貨商品，加入原補貨單沒有的任意規格。
2. 原品項和追加品項可在送出前調整；同一 SKU 不會重複出現。
3. 後端拒絕省略原品項、重複 SKU、停用 SKU、無效數量、無效日期及非採購角色。
4. 確認後追加品項以同一補貨單號寫入，申請量 0、下單量正確，且操作人與操作紀錄完整。
5. LINE 下單通知與到貨確認頁都包含追加品項。
6. 重送相同操作不新增第二批追加列。
7. 既有只調整原申請品項的流程不退化。
8. 指定測試、完整測試、lint 與 build 全部通過。
