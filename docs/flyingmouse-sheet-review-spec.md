# 規格：飛鼠目錄差異審核與新品匯入

## 目標

把每日飛鼠官方 Excel 差異寫入 Google Sheet 的獨立審核分頁，讓人員先看過內容、明確選擇「核准匯入」後，程式才把新品加入 `SKU主檔`。

此階段只自動匯入「飛鼠存在、`SKU主檔` 尚未存在」的新品。描述變更、來源缺漏與衝突只列入審核，不自動修改或刪除既有 SKU。流程仍不會寫入飛鼠後台。

## 使用者流程

1. Cloud Run Job 下載並驗證飛鼠官方 Excel。
2. 以 `SKU主檔!A:D` 產生差異。
3. 將差異以固定同步鍵 upsert 至 `飛鼠目錄待確認`。
4. 人員在 `審核狀態` 選擇：
   - `待確認`：不處理。
   - `核准匯入`：下一次 Job 重新驗證來源後匯入新品。
   - `略過`：保留記錄但不匯入。
5. 匯入成功後，系統將狀態改為 `已匯入`；若來源內容在核准後改變，改為 `需重新確認`，不寫入主檔。

## 審核分頁資料契約

分頁名稱固定為 `飛鼠目錄待確認`，欄位順序固定：

| 欄 | 欄位 | 寫入者 | 說明 |
| --- | --- | --- | --- |
| A | 同步鍵 | 系統 | 差異類型與 SKU 組成的穩定唯一鍵 |
| B | 偵測狀態 | 系統 | `待處理` 或 `已解除` |
| C | 差異類型 | 系統 | `新增`、`描述變更`、`來源缺漏`、`衝突` |
| D | 審核狀態 | 人員／系統 | `待確認`、`核准匯入`、`略過`、`已匯入`、`需重新確認` |
| E | SKU | 系統 | 飛鼠貨品編號或既有 SKU |
| F:K | 飛鼠資料 | 系統 | 商品名稱、兩層規格、庫存、GTIN、儲位 |
| L:N | 既有資料 | 系統 | `SKU主檔` 的商品名稱與兩層規格 |
| O | 來源指紋 | 系統 | SKU、名稱、規格、GTIN、儲位的 SHA-256；用於核准後重驗 |
| P | 首次偵測時間 | 系統 | 第一次看到此同步鍵的台北時間 |
| Q | 最後偵測時間 | 系統 | 最近一次仍存在的台北時間 |
| R | 審核備註 | 人員 | 自由文字，系統不得覆寫 |
| S | 處理時間 | 系統 | 解除、匯入或要求重新確認的台北時間 |

系統更新既有列時不得覆寫 `審核備註`。除來源指紋改變使既有核准失效外，也不得重設人員已選擇的審核狀態。

## 新品匯入規則

只有同時符合下列條件的列可以匯入：

- `偵測狀態 = 待處理`
- `差異類型 = 新增`
- `審核狀態 = 核准匯入`
- SKU 仍存在於當次已驗證的飛鼠官方 Excel
- 審核列的來源指紋與當次來源身分欄位一致
- `SKU主檔` 沒有同一 SKU 的不同資料

寫入 `SKU主檔` 時只碰非公式欄：

| 主檔欄位 | 值 |
| --- | --- |
| A:G | 飛鼠 SKU、商品名稱、規格、庫存、GTIN、儲位 |
| H | 商品名稱與非空規格以 `｜` 串接 |
| I | `一般SKU` |
| J | `是`，核准匯入後立即出現在可補貨清單 |
| K | 不寫入，沿用既有陣列公式 |
| L | `件` |
| M | `飛鼠` |
| N | 不寫入，沿用既有陣列公式 |

`SKU主檔` 目前允許的下拉值已由正式表驗證：I 欄含 `一般SKU`，J 欄含 `是`。若表頭不符、主檔已滿、SKU 重複或核准已過期，整批主檔寫入必須安全停止。

庫存不納入核准指紋，因為它是高頻變動值；核准匯入時一律使用當次官方 Excel 的最新庫存快照。商品名稱、規格、GTIN 或儲位有任何變動時，舊核准仍會失效。

## 可重跑與併發保護

- 同一差異使用固定同步鍵，不重複新增審核列。
- 不刪除已消失的差異列，只把 `偵測狀態` 改為 `已解除`。
- 已存在且內容相同的 SKU 視為冪等成功，可把審核列標為 `已匯入`，不可再新增一列。
- 每次寫入前重新讀取審核分頁與 `SKU主檔`。
- Google Sheets 寫入使用明確範圍；主檔與審核狀態盡量放在同一個 `values.batchUpdate`。
- Cloud Run Job 維持單一 task；排程不與 LINE Web Service 共用容器。

## 失敗保護

- 審核分頁既有表頭不符時停止，不猜測欄位。
- 來源 Excel 驗證失敗時不得更新審核分頁。
- 來源指紋改變時把 `核准匯入` 改為 `需重新確認`，不得沿用舊核准。
- `描述變更`、`來源缺漏`、`衝突` 不得寫入 `SKU主檔`。
- 不覆寫 K、N 陣列公式，不自動把新品設為可補貨。
- 不記錄飛鼠帳密、Cookie 或瀏覽器儲存狀態。

## 驗收標準

- 零差異執行可建立並格式化審核分頁，但不修改 `SKU主檔`。
- 新品差異會新增一列，重跑只更新最後偵測時間。
- 人工備註與未過期的審核狀態在重跑後保留。
- 核准後來源有變時不匯入，狀態改為 `需重新確認`。
- 合格核准只新增一筆主檔資料，I/J 預設為 `一般SKU`／`是`，K/N 不被寫入；核准完成後可立即在 LINE 補貨清單使用。
- `npm test`、`npm run lint`、`npm run build` 與依賴稽核全部通過。
